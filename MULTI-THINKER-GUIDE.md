## 📊 详细对比

### multi-thinker-debate（对抗模式）

**文件**：`multi-thinker-debate.md`
**角色**：进取派 vs 怀疑派 vs 务实派
**适用场景**：
- ✅ 架构决策（微服务 vs 单体）
- ✅ 技术选型（TypeScript vs JavaScript）
- ✅ 风险评估（这个方案会爆炸吗？）
- ✅ 性能优化（ROI 评估）

**典型提问**：
```bash
"三方辩论：是否采用微服务"
"批判性分析：这个架构有致命缺陷吗"
"React vs Vue，帮我决策"
```

**输出示例**：
```
🚀 进取派：必须迁移！性能提升 3 倍
🛡️ 怀疑派：分布式复杂度会爆炸
⚖️ 务实派：团队没人会 K8s

→ 综合仲裁：模块化单体（折中方案）
```

## ⚠️ 常见误区

### ❌ 误区 1：用辩论模式做脑暴
```
错误："三方辩论：设计一个登录功能"
问题：登录功能设计需要创意，不需要对抗
正确："脑暴一下：登录功能如何创新"
```

### ❌ 误区 2：用脑暴模式做决策
```
错误："脑暴一下：是否重写项目"
问题：重写是明确的 Yes/No 决策，需要风险评估
正确："三方辩论：该不该重写项目"
```

### ❌ 误区 3：混淆目标
```
辩论模式目标：做选择（A vs B vs C）
脑暴模式目标：找可能性（A + B + C + ...）

决策前先脑暴，脑暴后再辩论
```

## 💡 使用技巧

### 技巧 1：组合使用
```
第 1 步：multi-thinker-brainstorm（发散）
  "脑暴一下微服务的各种可能方案"
  → 产生 20+ 个想法

第 2 步：multi-thinker-debate（收敛）
  "三方辩论：选择哪个方案"
  → 从 20 个中选出最优解
```

### 技巧 2：根据问题类型选择
```
开放式问题（How/What） → 脑暴模式
  "如何改进用户体验？"
  "有什么创新点？"

封闭式问题（Whether/Which） → 辩论模式
  "是否采用 X？"
  "哪个更好：A 还是 B？"
```

### 技巧 3：调整角色权重
```
辩论模式：
  "务实派权重最高"（成本敏感项目）
  "进取派权重最高"（创新项目）

脑暴模式：
  "艺术家 + 心理学家权重高"（用户体验）
  "科学家 + 工程师权重高"（技术方案）
```
